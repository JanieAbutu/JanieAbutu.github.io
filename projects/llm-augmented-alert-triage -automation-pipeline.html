LLM-Augmented SOC Automation Pipeline

A lightweight SOAR-style pipeline built to automate SOC enrichment, severity scoring, and analyst context generation using deterministic workflows and AI-assisted analysis.

The Problem

Security Operations Centers process large volumes of alerts daily, but many alerts lack sufficient context for rapid analyst decision-making.

Analysts often spend significant time manually:

  • Enriching indicators with threat intelligence
  • Assigning severity levels
  • Documenting incidents
  • Correlating investigation context

This creates operational problems including alert fatigue, inconsistent triage outcomes, slower response times, and limited auditability.

The goal was to design an AI-assisted SOC pipeline where security decisions remain deterministic and auditable.

What I Built

I built a lightweight SOC automation pipeline using n8n that simulates how modern SOC and SOAR environments automate enrichment, classification, contextual analysis, and incident storage.

  • Webhook-based alert ingestion
  • VirusTotal threat intelligence enrichment
  • Rule-based severity scoring
  • LLM-powered analyst summaries via Ollama
  • MITRE ATT&CK mapping
  • Structured incident aggregation
  • PostgreSQL audit logging

The architecture intentionally separates deterministic security logic from AI-generated contextual analysis.

How It Works

  1. Simulated alerts are generated and sent into an n8n webhook workflow.
  2. Indicators such as IP addresses and domains are extracted for enrichment.
  3. VirusTotal retrieves threat intelligence signals including malicious detections and reputation data.
  4. A rule-based JavaScript engine assigns:
    • Severity level
    • Confidence score
    • Recommended response action
  5. Ollama generates analyst-style summaries and MITRE ATT&CK mappings.
  6. A final data aggregation layer merges raw alerts, VirusTotal enrichment, rule-based severity decisions, MITRE ATT&CK mappings, and AI-generated contextual analysis into a single structured SOC incident object before storage.
  7. Final incident objects are stored in PostgreSQL for auditing, reporting, dashboards, and future reference.

Engineering Approach

Many AI-assisted security products centralize decision-making inside the model itself. I intentionally avoided that design pattern.

The system was designed around several principles:

  • Security decisions should remain deterministic
  • Analysts require explainable and auditable workflows
  • AI should augment investigations rather than replace detection logic
  • Rule engines provide predictable and consistent outcomes

This architecture reflects how real SOC pipelines separate:

  • Ingestion
  • Enrichment
  • Decisioning
  • Contextual analysis
  • Orchestration
  • Storage and auditing

Impact

  • Reduced repetitive manual triage tasks through automation
  • Improved incident consistency using deterministic scoring logic
  • Enhanced analyst visibility with AI-generated context
  • Created structured audit-ready incident storage
  • Built a modular architecture capable of scaling with additional integrations

The project demonstrates how AI can responsibly augment SOC operations without replacing core security decision-making processes.
